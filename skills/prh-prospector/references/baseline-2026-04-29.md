# Baseline-mittaus 2026-04-29

Ensimmäinen baseline-ajo PRH-Prospector v1:llä. 28 yritystä, 3 yritystä per
target. Tarkoitus: 4-6 viikon kuluttua aja sama otos uudelleen ja vertaa
Vainu-mittauksen kanssa.

Ajot tallennettu polkuun `/var/lib/paperclip/prh-prospector/baseline/`:
- `alli-audit-2026-04-29.json` (aggregaatti)
- `alli-audit/{Y-tunnus}.jsonl` (yksityiskohdat per yritys)
- vastaava `saatavilla/`, `ololla/`

## Yhteenveto v1.0 → v1.1

v1.1-parannukset ajettu samalle otokselle samana päivänä:
- Realistinen browser User-Agent (Mozilla/5.0 Chrome/121)
- TLD-tunnistus yritysnimessä (Verkkokauppa.com → suoraan oikea URL)
- `--prefer postal` -käyttäytyminen kun käyntiosoitetta ei ole
- Booking.com-fallback hospitality-TOL-koodeille (551*, 552*, 553*, 791*, 792*, 799*)
- Facebook-sivu-arvaus (yleensä blocked, mutta yrittää)

## Tulkinta

### Alli-Audit (isot EAA-velvolliset)
- Erinomainen PRH-perusdatan kattavuus (100 %) — odotettavissa, koska kaikki
  ovat suuria listattuja/tunnettuja yrityksiä.
- Verkkosivu-löytyvyys 50 % — vain puolet on löydettävissä yksinkertaisella
  `<yritysnimi>.fi/.com`-arvauksella. Esim. Verkkokauppa.com:in slug-pelkistys
  poisti `.com`-osan ja päätyi väärään URL:iin. Posti ja Gigantti palautti
  HTTP-virheen botin tunnistuksen takia.
- **Korjausehdotus**: lisää `--url=...`-argumentti pakolliseksi tunnetuille
  yrityksille tai integroi Google-haku slug-arvauksen ohelle.

### Saatavilla (pienet parturit-kampaamot)
- PRH-data 100 %, mutta verkkosivu-löytyvyys vain 10 %. Useimmat pienet
  parturit/kauneushoitolat eivät ole verkossa tai käyttävät Facebook-sivua
  virallisena verkkopresenssina.
- **Vainu-vertailu kriittinen**: jos Vainu löytää näille tech-stackin/somen,
  se on Vainun pää-arvonlähde tässä segmentissä.

### Ololla (hotellit/majoitus)
- PRH-data 100 %, mutta käyntiosoite kummallisesti 0 % — ololla-otoksessa
  vanhempia hotelleja, joiden osoitteet on rekisteröity vain postiosoitteena.
  **Vaatii lisätutkimusta**: ehkä client-side-yhdistäminen postiosoite ↔
  käyntiosoite tarvitaan jos PRH ei vain palauta visiting-tyyppiä.
- Verkkosivu-löytyvyys 20 % — hotellit usein toimivat
  Booking.com/Airbnb-kautta ilman omaa sivua, tai sivu on sub-domain
  (esim. parent-yhtiön alla).

## Vainu-vertailusuunnitelma 6 viikon päähän (2026-06-10)

1. **Aja sama 28 yrityksen otos uudelleen** PRH-Prospectorilla
   (tämän skriptin ajaminen ja tulosten vertailu)
2. **Pyydä Vainu-pilotti** mukaan saman otoksen mittaukseen
3. **Mittaa kattavuusero per Vainu-namespace**:
   - `financial_statements.*` (104) — Vainu täyttänee, PRH ei v1:ssä
   - `social_media.*` (6) — verrattava Vainu vs PRH-AI
   - `website_data.*` (7), `technology_data.*` (5) — Vainun pää-moatti
   - `recruitment_keywords.*` (6) — vainu-tärkeä signaali
   - `vainu_custom_industries.*` (6) — Vainun kvalitatiivinen taggaus
4. **Päätös**:
   - PRH-Prospector kattavuus ≥ 80 % painotettu Vainun arvosta → **älä jatka
     Vainua**, laajenna PRH-skilliä
   - PRH-Prospector ≈ 50 % → **jatka Vainua vain Alli-Auditille**, käytä
     PRH:ta Saatavilla/Olollalla
   - Signaalit kriittisiä → **Vainu pysyy ainoana** ratkaisuna

## Tunnistetut puutteet — v1.1 (2026-04-29)

### Toteutetut
- [x] Slug-arvaus puhdistaa `.com`-päätteen liian aggressiivisesti — TLD-tunnistus nimessä lisätty (Verkkokauppa.com toimii)
- [x] Botin tunnistus 401/403-vasteet — realistinen Chrome/121 User-Agent + Accept-Language fi-FI lisätty (Posti onnistuu nyt)
- [x] Käyntiosoite vs postiosoite — fallback postal-tyyppiin kun visiting puuttuu
- [x] Facebook-sivu-arvaus PRH:n nimellä — toteutettu (mutta Facebook estää useimmat HEAD-pyynnöt 400:lla)
- [x] Booking.com-fallback hospitality-TOL-koodeille — URL-pattern lisätty (URL toimii vain jos hotelli on rekisteröity tarkasti tällä slugilla)

### Jäljellä v1.2:ksi

- [ ] Sivun footer-fetching jos etusivun sosiaalinen media löytyy harvemmin (HTML truncation 250KB cuttaa pitkät sivut, footer voi jäädä pois)
- [ ] Booking.com-haku oikealla parametrilla (search API tai web-scrape hotellin nimellä)
